## 1. 产品概述

### 1.1 一句话描述

**DropShipFlow** 是一款面向跨境电商卖家的 SaaS 工具，用户在 1688 浏览商品时，只需一键点击即可完成「抓取 → AI 翻译 → 图像重处理 → 智能定价 → 上架到 Shopify」的全自动化流程，将传统 30-60 分钟的手动上架工作压缩到 30 秒。

### 1.2 核心价值主张

> 把 1688 上的任何一个商品，在 3 分钟内变成 Shopify 店铺里一个可直接销售的、英文优化过的商品。

**关键词：** 一键、自动化、AI 驱动、零门槛。

## 2. 市场背景与问题分析

### 2.1 市场现状

- **跨境电商 Dropshipping 市场规模**：2025 年全球约 $3500 亿，年增长率 ~23%
- **Shopify 活跃店铺数**：超过 400 万家
- **1688 作为货源**：是中国跨境卖家最主流的货源平台，SKU 数量亿级

### 2.2 用户痛点（按痛苦程度排序）

| 优先级 | 痛点 | 详细描述 | 当前解决方案 | 痛苦程度 |
|--------|------|---------|-------------|---------|
| P0 | **手动上架效率极低** | 从 1688 复制标题/描述/图片/SKU 到 Shopify，逐个填写，单个商品耗时 30-60 分钟 | 纯手工 / 雇佣 VA（虚拟助理） | ★★★★★ |
| P0 | **翻译质量差** | 用 Google 翻译出来的英文商品描述不专业、不通顺，导致转化率低 | Google Translate / ChatGPT 手动复制粘贴 | ★★★★★ |
| P1 | **图片处理繁琐** | 1688 图片有水印、尺寸不统一、缺乏英文营销元素，直接使用会降低品牌形象 | Photoshop 手动修图 / 外包 | ★★★★☆ |
| P1 | **定价计算复杂** | 需要考虑汇率、运费、平台佣金、广告成本、利润空间等多个变量，手工计算易出错 | Excel 表格 / 凭感觉 | ★★★★☆ |
| P2 | **SKU 变体管理** | 1688 商品的规格（颜色/尺寸等）组合复杂，手动映射到 Shopify Variants 极易出错 | 逐个手动创建 | ★★★☆☆ |
| P2 | **多店铺管理困难** | 有多个 Shopify 店铺的卖家需要重复操作 | 分开操作 / 不做了 | ★★★☆☆ |

### 2.3 竞品分析

| 竞品 | 定位 | 优势 | 劣势 | 我们的差异化 |
|------|------|------|------|-------------|
| **Oberlo** (已关闭) | Shopify 官方 Dropshipping 插件 | 与 Shopify 深度集成 | 已停止服务 | — |
| **DSers** | AliExpress → Shopify | 免费，官方合作 | 只支持 AliExpress，不支持 1688 | 专注 1688 货源 |
| **CJDropshipping** | 综合货源平台 | 自有供应链 | 加价高，商品有限 | 用户自选 1688 商品，灵活度高 |
| **AutoDS** | 多渠道自动上架 | 功能全面 | 贵（$26.9/月起），对 1688 支持弱 | 专注 1688 → Shopify 单一链路，价格更低 |
| **各种 ERP (店小秘/马帮)** | 跨境 ERP | 功能全面 | 面向大卖家，复杂，贵 | 轻量、专注、AI 原生 |

**结论：** 市场上没有一个产品专注做「1688 → Shopify」这条单一链路的一键 AI 自动化。这是一个清晰的市场空白。

## 4. 产品愿景与定位

### 4.1 产品愿景

成为跨境电商卖家的「商品上架操作系统」—— 任何人在任何地方发现一个好产品，都能在 1 分钟内让它出现在自己的 Shopify 店铺里，并且看起来就像品牌原生的商品。

### 4.2 产品定位

**一句话定位：** 最智能的 1688 → Shopify 一键上架工具。

**品牌调性：**
- 简单：一键操作，无需学习
- 智能：AI 自动处理翻译、图像、定价
- 可靠：数据准确，不漏不错
- 专业：输出的商品看起来像品牌原生

### 4.3 商业模式

| 层级 | 价格 | 核心功能 |
|------|------|---------|
| **Free** | $0/月 | 10 个商品/月，基础翻译，1 个店铺，带水印 |
| **Pro** | $29/月 | 200 个商品/月，AI 翻译，去水印，自定义定价规则，3 个店铺 |
| **Business** | $99/月 | 1000 个商品/月，优先处理，团队协作，10 个店铺，API 接入 |
| **Enterprise** | 定制 | 无限商品，私有化部署，专属支持 |

## 6. 用户故事

### 6.1 核心用户旅程

#### 故事 1：新手卖家第一次使用

> **小李**刚开了一个 Shopify 店铺卖家居用品。他在 1688 上发现了一款很有潜力的收纳盒。他安装了 DropShipFlow 插件，点了一下按钮，3 分钟后，这个收纳盒就出现在了他的 Shopify 后台草稿箱里——标题是地道的英文，图片没了水印还加了白底，价格也自动算好了（成本 ¥28 → 售价 $11.99）。他检查了一下觉得没问题，点了发布。全程不到 5 分钟。

#### 故事 2：成熟卖家批量操作

> **小王**有 3 个 Shopify 店铺，分别卖家居、户外和电子配件。他每天在 1688 上选 50 个新品。用 DropShipFlow，他批量浏览商品，逐个点击抓取，后台自动排队处理。他预先设置了 3 套定价规则（家居 3.5 倍加价、户外 4 倍、电子 2.8 倍），商品会自动匹配对应店铺的定价规则。1 小时后，50 个商品全部在对应店铺的草稿箱里等着他审核。

#### 故事 3：VA 团队协作

> **某代运营团队**管理了 15 个客户的 Shopify 店铺。运营助理负责在 1688 选品和抓取，运营经理负责审核翻译和定价，客户最终确认后发布。DropShipFlow 的团队协作功能让这个流程完全在线化——助理提交、经理审核、客户确认，每个环节都有通知。

#### 故事 4：超级管理员管理多个团队

> **老张**是 DropShipFlow 的第一个注册用户，自动成为超级管理员。他创建了两个团队：A 团队（Pro 套餐，10 人）和 B 团队（Free 套餐，2 人），各自指定了管理者。他自己不属于任何团队，但可以随时切换到任意团队查看数据、协助排查问题。A 团队成员能看到去水印和营销主图功能，B 团队菜单里没有这些选项。

## 7. 功能详细规格

### 7.0 权限与团队模型

#### 7.0.1 核心设计原则

| 原则 | 说明 |
|------|------|
| **一个用户 = 一个团队** | 用户注册时必须归属于某个团队，不能同时在多个团队 |
| **超级管理员是全局角色** | 不属于任何团队，可进入任意团队查看/操作所有数据 |
| **团队 = 数据隔离边界** | 店铺、商品、定价规则、导入任务全部归属团队，跨团队不可见 |
| **套餐 = 功能权限边界** | Free/Pro/Business 对应不同的菜单和能力集 |
| **管理者 = 团队内管理员** | 可管理团队成员和店铺绑定，普通成员不可 |

#### 7.1.1 触发方式

#### 7.1.3 抓取策略

```
策略 1（默认）：直接 DOM 提取
  - 优点：速度最快
  - 缺点：1688 改版可能失效

策略 2（降级）：MutationObserver 监听 + 滚动触发
  - 优点：兼容懒加载内容
  - 缺点：需要等待渲染完成

策略 3（逃生舱）：截取页面截图 + 多模态模型识别
  - 优点：完全不依赖 DOM 结构
  - 缺点：成本高、速度慢
```

**DOM 选择器采用远程配置：** 插件启动时从服务端拉取最新选择器规则 JSON，1688 改版只需后台更新配置，无需发版。

#### 7.1.5 登录流程

插件**不内置**登录表单（380px 的小窗口输入账号密码体验太差）。采用 **「Web 端登录 → 自动回传 Token 给插件」** 方案：

```
┌──────────────┐        ┌─────────────────┐        ┌──────────────┐
│  浏览器插件   │        │  Web 管理后台     │        │  Chrome     │
│  (Popup)     │        │  admin.dropship  │        │  Storage    │
│              │        │  flow.com        │        │  API        │
└──────┬───────┘        └────────┬────────┘        └──────┬───────┘
       │                         │                        │
       │ ① 用户点击"登录"按钮     │                        │
       │ chrome.tabs.create()   │                        │
       │ 打开 admin.../login    │                        │
       │ ──────────────────────▶│                        │
       │                         │                        │
       │                         │ ② 用户输入邮箱/密码     │
       │                         │ POST /auth/login      │
       │                         │ 返回 JWT Token        │
       │                         │                        │
       │ ③ 登录成功 — Web 端通过 │                        │
       │ chrome.runtime         │                        │
       │ .sendMessage() 发送     │                        │
       │ Token 给插件            │                        │
       │ ◀──────────────────────│                        │
       │                         │                        │
       │ ④ 插件接收并存储 Token   │                        │
       │ ──────────────────────────────────────────────▶│
       │                         │                        │
       │ ⑤ 此后每次打开 Popup    │                        │
       │ 自动从 Storage 读取     │                        │
       │ ◀──────────────────────────────────────────────│
       │ Token，无需再次登录     │                        │
```

**实现要点：**

| 要点 | 说明 |
|------|------|
| **通信通道** | `chrome.runtime.sendMessage` + `externally_connectable` 白名单 |
| **Token 存储** | `chrome.storage.local`（插件专属，其他网站/插件无法访问） |
| **Token 刷新** | 15 分钟过期，用 Refresh Token 静默刷新，用户无感 |
| **退出登录** | 清除 `chrome.storage.local` 中的 Token 即可 |
| **安全** | Token 通过 Chrome 内部消息通道传递，不经过网络，不暴露给网页 DOM |

### 7.4 定价规则引擎

#### 7.4.1 定价公式

```
SalePrice = roundTo(
  (SourcePriceCNY / ExchangeRate) × MarkupMultiplier,
  rule.round_to
)

CompareAtPrice = SalePrice × (1 + compare_at_markup_percent)
Cost = SourcePriceCNY / ExchangeRate
Margin = SalePrice - Cost
```

#### 7.4.2 规则示例

```yaml
规则 1: "家居品类低客单价"
  条件:
    - source_price_range: 0 - 50 CNY
    - category_contains: "家居"
  公式:
    exchange_rate: 7.25
    markup_multiplier: 3.5
    round_to: ".99"
    compare_at_markup: 0.5  # 划线价比售价高 50%
  
规则 2: "家居品类高客单价"
  条件:
    - source_price_range: 50 - 999999 CNY
    - category_contains: "家居"
  公式:
    exchange_rate: 7.25
    markup_multiplier: 2.5
    round_to: ".99"

规则 3: "电子品类"
  条件:
    - category_contains: "电子"
  公式:
    exchange_rate: 7.25
    markup_multiplier: 2.2
    round_to: ".95"

规则 4: "默认兜底规则"
  条件:
    - source_price_range: 0 - 999999 CNY  # 匹配所有
  公式:
    exchange_rate: 7.25
    markup_multiplier: 3.0
    round_to: ".99"
```

**匹配逻辑：** 按规则优先级从上到下匹配，命中第一个符合条件的规则即停止。默认规则放在最后。

| **只记录变更，不记录查询** | GET/列表类请求不产生日志，避免噪音和存储膨胀 |
| **全自动，零侵入** | 通过中间件/ORM 钩子自动捕获，业务代码无需显式调用 |
| **记录变更前后值** | `old_values` 和 `new_values` 以 JSONB 存储，支持 diff 对比 |
| **不可篡改** | 日志表只允许 INSERT，不允许 UPDATE/DELETE |
| **按团队隔离查询** | 超管看全部，管理者看本团队，成员无权限 |

#### 7.7.2 记录范围

### 7.10 在线监控与强制下线（超管功能）

#### 7.10.1 实现方案

不新建数据库表，用 **Redis** 做会话追踪 + **JWT 黑名单** 做强制下线。

```
每次 API 请求:
  → 中间件检查 Redis 黑名单: token.jti 是否在黑名单中?
      → 是 → 401 "您的会话已被管理员终止"
      → 否 → 刷新 session 续期 → 放行

超管踢人:
  → Redis DEL active_session:{user_id}
  → Redis SADD jwt_blacklist {token.jti} (TTL = token 剩余有效期)
  → 用户下次请求直接被 401
```

### 7.11 AI 提供商管理（超管配置）

#### 7.11.1 设计

超管可随时新增、切换、禁用 AI 提供商。系统自动按优先级降级。翻译和图像生成可使用不同提供商。

``` 
ai_providers 表:
  ├── name: "DeepSeek" / "GPT-4o" / "Gemini Flash" 等
  ├── category: "text" (文字AI) | "vision" (视觉AI)
  ├── provider_type: deepseek / openai / anthropic / google
  ├── api_base_url: API 端点
  ├── api_key: 加密存储
  ├── default_model: 默认模型名称
  ├── available_models: ["deepseek-chat", "deepseek-reasoner", ...]
  ├── is_default: 是否该类别下的默认 (每类别只有一个可为 true)
  ├── priority: 降级优先级 (数字越小越优先，仅在同 category 内比较)
  ├── max_retries / timeout_seconds
  └── pricing: {input_per_1k, output_per_1k} — 成本核算

设计要点:
  · Text 和 Vision 各自维护独立的优先级链，互不干扰
  · 翻译任务走 text 链: DeepSeek → (失败) → OpenAI → (失败) → Claude
  · 截图识别/图像理解走 vision 链: Gemini Flash → (失败) → GPT-4o → (失败) → Claude Opus
  · 超管可以为 text 和 vision 配置完全不同的提供商组合
```

| Shopify API Key 存储 | AES-256 加密存储 |
| 数据传输 | HTTPS/TLS 1.3 |
| 图片存储访问控制 | 签名 URL（带过期时间） |
| SQL 注入防护 | ORM 参数化查询 |
| XSS 防护 | 输入过滤 + CSP Header |

### 8.4 可扩展性

- **插件 DOM 选择器：** 远程配置，不依赖插件发版更新
- **翻译模型：** 支持切换不同 LLM Provider（DeepSeek / OpenAI / Claude）
- **ComfyUI Workflow：** 新增处理步骤无需改代码，上传新 workflow JSON 即可
- **定价规则：** 完全配置化，支持动态新增规则
- **多平台扩展：** 架构预留了支持其他货源平台（淘宝、拼多多）和其他电商平台（WooCommerce、Amazon）的接口

## 11. 风险与依赖

### 11.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 1688 反爬升级 | 插件抓取失效 | 高 | DOM 选择器远程配置 + 多模态截图降级方案 |
| Shopify API 变更 | 同步失败 | 中 | 版本化 API 调用 + 变更监控 |
| ComfyUI GPU 不稳定 | 图片处理积压 | 中 | 多 Worker 冗余 + 任务重试队列 |
| LLM API 服务中断 | 翻译失败 | 低 | 多 Provider 备用（DeepSeek → OpenAI → 本地模型） |
| 1688 图片防盗链 | 图片无法下载 | 中 | 通过插件下载（带 Referer）而非服务端下载 |

### 11.2 业务风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 1688 封禁账号 | 用户无法选品 | 低 | 插件模拟正常用户行为，非高频抓取 |
| 法律合规 | 侵权/售假 | 中 | 敏感词检测 + 用户协议免责声明 + 侵权举报通道 |
| 竞品跟进 | 价格战 | 高 | 持续迭代 AI 能力（图像处理差异化），建立品牌 |

### 11.3 外部依赖

| 依赖 | 影响范围 | 降级方案 |
|------|---------|---------|
| 1688 网站结构 | 抓取功能 | DOM 多策略 + 截图模式 |
| Shopify Admin API | 商品上传 | 导出 CSV 手动导入（逃生） |
| DeepSeek API | AI 翻译 | 切换 OpenAI / Claude |
| ComfyUI | 图像处理 | 跳过处理 + 标记待处理 |
| Cloudflare R2 / S3 | 图片存储/CDN | 本地存储 + Nginx 代理 |
